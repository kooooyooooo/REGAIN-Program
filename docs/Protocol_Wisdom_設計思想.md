# REGAIN_Protocol と REGAIN_Wisdom 設計思想メモ

> このドキュメントは、REGAINプログラムのコンテンツ作成における根本思想を言語化したものです。

---

## 1. 概念の定義

### REGAIN Wisdom（叡智）

| 項目 | 内容 |
|------|------|
| **領域** | Mindset / Theory |
| **役割** | なぜやるのか、どう考えるべきか、AIとどう向き合うか |
| **価値** | 思考のOSをアップデートする |

AIに指示を出して処理を待っている間に、コーチが持つ**思想・フレームワーク・マインドセット**を伝えるコンテンツ。

- 「なぜそれをやるのか（WHY）」を腹落ちさせる
- 待ち時間を「思考が整理される時間」に変換する
- 次回以降自走するための判断基準を渡す

#### なぜ「Teaching」ではなく「Wisdom」なのか

| Teaching（ティーチング） | Wisdom（ウィズダム） |
|------------------------|---------------------|
| 受動的 | 能動的 |
| 一過性の情報 | 蓄積される資産 |
| 教える側 → 教わる側 | 先人 → 継承者 |
| 「正解」を与える | 「判断基準」を渡す |
| 消費される | 受け継がれる |

**「Teaching」は消費される。「Wisdom」は資産として残る。**

REGAINが目指すのは「教育」ではなく「知恵の伝承」。受講者は単なる「生徒」ではなく、知恵を受け継ぎ、自らの現場で応用し、やがて次の誰かに伝える「継承者」である。

この命名には、受講者を受動的な学習者から**能動的な実践者・継承者**へと位置づけ直す意図がある。

---

### REGAIN Protocol（手順）

| 項目 | 内容 |
|------|------|
| **領域** | Action / Execution |
| **役割** | 具体的にどのボタンを押し、どう指示し、どう修正するか |
| **価値** | 迷いを消し、最短距離で成果物を生み出す |

受講者が実際に手を動かしながら体験的な学習をするための**具体的なステップバイステップガイド**。

- 「これをやれば必ずできる」を保証する
- 判断を求めず、実行に集中させる
- プロンプトはコピペ可能な形式で提供

---

## 2. WisdomとProtocolの対比

| 観点 | Wisdom（叡智） | Protocol（手順） |
|------|----------------|------------------|
| **領域** | Mindset / Theory | Action / Execution |
| **問い** | WHY（なぜやる） | HOW（どうやる） |
| **価値** | 思考のOSをアップデートする | 迷いを消し、最短距離で成果物を生み出す |
| **タイミング** | AIが処理中の待ち時間 | 受講者が手を動かす時間 |
| **形式** | フレームワーク + 事例 | ステップバイステップ + プロンプト |
| **効果** | 明日以降も自走できる | 今日の成果物が完成する |
| **比喩** | 釣り方を伝承する | 魚を一緒に釣る |

## 3. Protocol作成の設計原則

### 3.1 「逃げ道を塞ぐ」設計

REGAINの本質は「成果物が完成すること」。Protocolはそのための強制力。

- **具体的なプロンプトまで書く** → 「何を入力すればいいかわからない」を排除
- **ステップを細分化する** → 「どこから始めればいいかわからない」を排除
- **選択肢を限定する** → 「選べない」を排除

### 3.2 「考えさせない」設計

- 受講者に判断を求めない
- 判断が必要な箇所は事前にWisdomで教える
- 「やれば動く」を保証する

### 3.3 「成功体験ファースト」

- まず動くものを見せる → その後に理屈を説明
- エラーが起きにくい設計にする
- 「自分でも作れた！」という感動を最優先

### 3.4 「再現性」の担保

- 誰がやっても同じ結果になるように書く
- 環境依存の部分は明記する
- ツールのバージョン・設定も必要に応じて記載

## 4. Wisdom作成の設計原則

### 4.1 「押しつけない」姿勢

- 「教える」ではなく「共有する」トーン
- 受講者の現場経験をリスペクトした語り口
- 「先輩からの助言」のようなスタンス

### 4.2 「事例ベース」の説得

- 抽象論だけでは腹落ちしない
- 成功例・失敗例をセットで見せる
- 受講者が「自分ごと化」できる事例を選ぶ

### 4.3 「フレームワーク」の提供

- 覚えやすい形で構造化（例：CPP = Context, Pain, Philosophy）
- 次回以降自分で使えるものを渡す
- 判断基準として機能するものを設計

### 4.4 「短く、濃く」

- AIの待ち時間は数分程度
- 要点を絞って伝える
- 長文より、刺さる一言

### 4.5 「現場の言葉」で書く

- 技術用語を避ける
- スポーツ指導者が使う言葉で表現
- 「エンジニア向け」ではなく「現場向け」

## 5. ProtocolとWisdomの関係性

```
Wisdom（なぜやるか）
    ↓ 理解した上で
Protocol（どうやるか）
    ↓ 実行して
成果物（動くアプリ）
    ↓ 振り返って
Wisdomの内容が「自分の経験」として定着
```

**重要**: ProtocolとWisdomは独立したコンテンツではなく、**同じセッション内で相互補完する関係**にある。

## 6. セッションにおける時間配分イメージ

```
[セッション開始]
    ↓
[Wisdom] 今日のテーマの「なぜ」を共有（5分）
    ↓
[Protocol] Step1 実行（受講者が手を動かす）
    ↓
[Wisdom] AIの待ち時間に補足知識を共有
    ↓
[Protocol] Step2 実行
    ↓
[Wisdom] 待ち時間にフレームワーク紹介
    ↓
[Protocol] Step3 実行 → 成果物完成
    ↓
[振り返り] 今日のWisdomを体験に紐づけて定着
```

## 7. 根底にある思想（COREBOOKより）

> REGAINとは、AIを教える事業ではなく、思考を放置してきた現場人を強制的に「考える側」に引き戻し、その結果として"作れる自分"を取り戻させる**再起動装置**である。

この思想において：

- **Protocol** = 実行の強制力を提供（逃げ道をなくす）
- **Wisdom** = 考える枠組みを提供（思考の整理）

両輪があって初めて「再起動」が完了する。

## 8. コンテンツ作成時のチェックリスト

- [ ] ステップは十分に細分化されているか？
- [ ] プロンプトはコピペ可能な形式か？
- [ ] 受講者に不要な判断を求めていないか？
- [ ] 誰がやっても同じ結果になるか？
- [ ] 成功体験につながる設計か？

### Wisdom作成時

- [ ] 「なぜ」が明確に伝わるか？
- [ ] 事例は受講者が自分ごと化できるか？
- [ ] フレームワークは覚えやすい形か？
- [ ] 待ち時間内に伝えられる分量か？
- [ ] 現場の言葉で書かれているか？
- [ ] 押しつけがましくないか？
