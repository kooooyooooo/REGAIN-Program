# 🔄 改善サイクルのやり方

---

## 🎯 まずは結論から

Week2で身につけてほしい「改善サイクル」は、こんな感じです：

```
1. Plan   → 要求/要件定義書を作成（または更新）
2. Do     → GenSparkでアプリを構築・デプロイ(github + vercel)
3. Check  → デプロイしたアプリを実際に触って確認
4. Action → 修正点を洗い出して、1に戻る
```

シンプルでしょ？でも、**回し方にコツがある**んです。

## 🧠 AIの「記憶」の仕組み

実は、AIには**「覚えていられる量」に限界**があります。

### 📚 本棚のイメージ

AIの記憶を「本棚」だと思ってください。

- 最初は空っぽ
- あなたの指示や会話が「本」として入っていく
- でも、棚には**入る量が決まっている**

```
┌─────────────────────────────┐
│ [要件] [指示1] [指示2] [修正1] │  ← まだ余裕
│ [修正2] [修正3] [追加] [変更]  │  ← 埋まってきた
│ [修正4] [修正5] [修正6] [???]  │  ← 古いのが落ちる！
└─────────────────────────────┘
```

本棚がいっぱいになると、**古い本から落ちていく**。

つまり、場当たり的な修正を重ねると：
- 最初の要件を忘れる
- さっき直した内容も忘れる
- 結果、**同じバグが復活する**

## ✅ 正しい改善サイクルの回し方

### 🔑 ポイントは「まとめてから伝える」

❌ 気づいたらすぐ指示
✅ **まとめて整理してから指示**

### 📋 具体的なステップ

```
【Check】アプリを触って確認
    ↓
「あ、ここ直したい」「これも追加したい」
    ↓
【Action】修正点をメモに全部書き出す
    ↓
「本当に必要？」「優先度は？」を整理
    ↓
【Plan】要求・要件定義書を更新する
    ↓
【Do】更新した要求・要件定義書をAIに渡して再構築
```

### 💡 ここが大事！

**修正点を見つけても、すぐにAIに伝えない。**

一度「要件定義書」に落とし込んでから、改めてAIに渡す。

これが「改善サイクル」の正しい回し方です。
