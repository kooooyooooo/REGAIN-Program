# 🔄 改善サイクルの話

---

## 1. 🎯 Week2でやった改善サイクル、覚えてますか？

Week2で「改善サイクル（PDCA）」の回し方を練習しましたよね。
要求定義書を作って → GenSparkで構築して → 確認して → 修正点をまとめて、また最初に戻る。

**実はWeek3でも、やることはほぼ同じです。**

「え、じゃあ何が変わるの？」って話ですよね。
答えはシンプル。**使う道具が変わるだけ**。
GenSparkでやっていたことが、GitHub / Vercelに置き換わる。それだけなんです。

---

## 2. 📚 Week3の改善サイクル

改めて、Week3版のPDCAを見てみましょう。

| ステップ | Week2（GenSpark） | Week3（GitHub / Vercel） |
| ---------- | ------------------- | -------------------------- |
| **Plan** | 要求/要件定義書を作成・更新 | 要求定義書を更新 or 新規作成 |
| **Do** | GenSparkで構築・デプロイ | AIにプルリクエスト作成を指示 → マージ |
| **Check** | デプロイしたアプリを確認 | ページを更新して確認（自動デプロイ） |
| **Action** | 修正点を洗い出して1に戻る | 修正点を洗い出して1に戻る |

ね、**ほとんど変わらないでしょ？**

「Do」がGitHubのプルリクエスト＋マージに変わって、「Check」がVercelの自動デプロイ後にページを更新するだけ。
Week2の経験がそのまま活きます。安心してください。

---

## 3. 🏠 なぜ毎回「Plan」から始めるの？

ここ、Week3でいちばん大事な話です。

develop環境が動き出すと、**目の前にアプリがある状態**になります。
触ってみて「あ、ここ微妙だな」と思ったら、すぐ「ここ直して」と言いたくなる。

でも、ちょっと待ってください。

### Week2で学んだ「パッチワーク修正の罠」、覚えてますか？

場当たり的に「ここ直して」「ついでにこれも」を繰り返すと、AIが混乱する。
あの教訓、**動くアプリが目の前にある環境だと、もっとハマりやすい**んです。

GenSparkの時は「まだ作り途中」って感覚があったから、整理してから伝えられた。
でもVercelにデプロイされたものを見ると、「もう完成に近いじゃん」と感じる。
**その油断が、パッチワーク修正を誘発します。**

だからこそ、**必ずPlan（要求・要件定義）に立ち返る**。
「そもそも何を改善したかったのか」を整理してからDoに進む。
この習慣が、Week3の品質を大きく左右します。

## 4. 🛠️ 実際の流れ

実際にはこんな順番で回ります：

```
1. アプリを触って気になる点を洗い出す（Check）
       ↓
2. 修正点をまとめて整理する（Action）
       ↓
3. 要求定義書を更新 or 新規作成（Plan）
       ↓
4. AIにPR作成を指示 → マージ（Do）
       ↓
5. ページを更新して確認（Check）
       ↓
   → 2に戻って繰り返し
```

最初の1回だけPlanからスタートして、2周目以降は **Check → Action → Plan → Do → Check** の順番。

### 💡 ポイント

**「0から作り直してください」も立派な選択肢です。**

パッチを重ねるよりリセットした方が速いケースは多い。
要求定義書がしっかりしていれば、作り直しは怖くありません。
むしろ2回目の方が、速くて良いものが出来上がります。

---

## 5. 💡 まとめ

### 3つのポイント

1. ✅ **Week2と本質は同じ**。道具がGenSpark → GitHub / Vercelに変わるだけ
2. ✅ **必ずPlan（要求定義）から始める**。「直して」の前に「なぜ直すか」を整理
3. ✅ **0から作り直す勇気を持つ**。パッチワークより、リセットの方が速い場合も多い
