# Protocol / Wisdom 設計思想

## 概念定義

### Protocol（手順書）
- **目的**: HOW（どうやる）
- **タイミング**: 受講者が手を動かす時間
- **形式**: ステップバイステップ + コピペ可能なプロンプト
- **効果**: 今日の成果物が完成する

### Wisdom（知恵）
- **目的**: WHY（なぜやる）
- **タイミング**: AIが処理中の待ち時間
- **形式**: フレームワーク + 事例
- **効果**: 明日以降も自走できる

## セッション内の流れ

```
Wisdom（なぜ） → Protocol（どうやる） → 成果物 → Wisdomが経験として定着
```

## Protocol設計原則

1. **逃げ道を塞ぐ**: プロンプトまで書く、ステップを細分化
2. **考えさせない**: 判断を求めず実行に集中
3. **成功体験ファースト**: まず動かす → 後で理屈
4. **再現性**: 誰がやっても同じ結果

## Wisdom設計原則

1. **押しつけない**: 「教える」ではなく「共有する」
2. **事例ベース**: 成功例・失敗例をセット
3. **フレームワーク**: 覚えやすい形で構造化
4. **短く濃く**: 待ち時間は数分
5. **現場の言葉**: 技術用語を避ける

## REGAINの本質

> REGAINとは、思考を放置してきた現場人を強制的に「考える側」に引き戻し、"作れる自分"を取り戻させる**再起動装置**

- Protocol = 実行の強制力
- Wisdom = 考える枠組み

両輪で「再起動」が完了する。
