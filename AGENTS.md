# AGENTS.md

このファイルは、このリポジトリで作業する AI 開発エージェント向けの指示書です。
作業開始前に必ず最初にこのファイルを読んでください。
ユーザーからこのファイルと矛盾する明示的な指示があった場合は、ユーザー指示を優先してください。

**重要: このファイルは編集・削除禁止です。**

---

## 1. 基本方針

- このプロジェクトは、主に AI エージェントによる実装を前提としています。
- 実装前に、必ず仕様理解を優先してください。
- 不明点がある場合は、勝手な拡張をせず、現在ある資料から最も妥当な解釈で最小実装を行ってください。
- 複雑にしすぎず、まずはシンプルに動くものを作ってください。
- 日本語ユーザー向けプロジェクトとして扱ってください。

---

## 2. 参照順序（重要）

実装・修正・設計の前に、以下の順で必ず参照してください。

1. ユーザーがこの会話で直接添付したファイル
2. `docs/要求定義書*`
3. `docs/要件定義書*`
4. `docs/` 配下のその他ドキュメント
5. `README.md`
6. ソースコード全体

### ルール
- ユーザー添付ファイルがある場合は、それを最優先してください。
- `docs/要求定義書` と `docs/要件定義書` がある場合、必ず読んでから実装してください。
- 仕様が曖昧な場合は、既存コードよりも要求定義・要件定義を優先してください。
- 既存コードと仕様書が矛盾する場合は、仕様書に合わせる方向で判断してください。
- 大きな変更の前には、関連ファイルを十分確認してから着手してください。

---

## 3. 出力・コミュニケーション方針

- ユーザーへの説明は日本語で行ってください。
- コード内コメントも原則日本語で記述してください。
- 専門用語を使う場合でも、必要に応じて短く補足してください。
- 変更内容は、何を・なぜ変えたかが分かるように説明してください。
- エラーが起きた場合は、黙って進めず、原因と対応方針を説明してください。

---

## 4. 実装方針

- まず最小構成で完成させ、その後必要に応じて改善してください。
- 不要な抽象化、過剰な設計、将来のためだけの複雑化は避けてください。
- 既存の設計・命名・ディレクトリ構成がある場合は、できるだけ踏襲してください。
- 1ファイルが肥大化しすぎる場合は適切に分割してください。
- UI はモバイルファーストを意識してください。
- セキュリティに関わる値はハードコードしないでください。
- **ローカルでの動作確認は行わないでください。**
- **動作確認はホスティング後の環境で行う前提です。**
- **ローカルサーバー起動、ローカルプレビュー、ローカル疎通確認は不要です。**

---

## 5. GitHub / PR / コミット方針（重要）

### コミット主体
- コミット author は **必ず以下を使用してください**  
  - `AUTHOR_NAME:（ここに固定値を記載）`
  - `AUTHOR_EMAIL:（ここに固定値を記載）`
- 上記以外の author / email を使用してコミットしないでください。
- bot 名義や未許可の別アカウント名義で勝手にコミットしないでください。
- Vercel 連携を想定しているため、commit author email は必ず GitHub アカウントに紐づくものを使用してください。

### コミット前チェック
- コミット前に、変更内容を確認してください。
- エラーがある場合は、原因を確認してからコミットしてください。
- author / email が指定値と一致していることを確認してからコミットしてください。

### PR 作成
- PR 作成前に、変更内容の要約を整理してください。
- 仕様に対して何を満たしたかを簡潔にまとめてください。
- デプロイ関連エラーが出た場合は、そのまま押し切らず、author / email / 権限の問題を先に確認してください。

## 6. 技術スタック

このセクションはプロジェクトごとに更新してください。

### Frontend
- Next.js
- TypeScript
- Tailwind CSS

### Backend
- Supabase
- PostgreSQL

### Hosting / Deploy
- Vercel

### Auth
- Supabase Auth

### Notes

- **環境変数は Vercel 上でのみ管理してください。**
- APIキーや秘密情報はソースコードに埋め込まないでください。

## 7. 非機能要件の標準方針

- スマホ最適化を優先する
- レスポンシブ対応を行う
- RLS を前提にデータアクセスを設計する
- 認証は仕様書に従って実装する
- 本番環境と開発環境を意識したブランチ運用を行う

## 8. 生成・修正対象の優先順位

作業時は以下を意識してください。

1. 仕様理解
2. 最小実装
3. GitHub 反映
4. ホスティング後の動作確認
5. ドキュメント更新

実装だけして終わりにせず、必要なら以下も更新してください。

- `README.md`
- `docs/要件定義書*`
- `DEPLOYMENT.md`
- 関連する画面一覧やデータ構造メモ

---

## 9. ドキュメント更新ルール

以下に該当する場合、必要に応じてドキュメントも更新してください。

- 画面が増えた
- データ構造が変わった
- 環境変数が変わった
- デプロイ手順が変わった
- 認証や権限仕様が変わった

`DEPLOYMENT.md` が存在する場合は、デプロイ手順との整合性も確認してください。

---

## 10. 禁止事項

- このファイル（AGENTS.md）の編集・削除
- ユーザーが明示していない大規模リファクタ
- 無断で技術スタックを変更すること
- 無断で認証方式を変えること
- 無断で DB スキーマを破壊的変更すること
- APIキー、秘密情報、個人情報のハードコード
- ローカルでの動作確認
- 動作未確認のまま「完了」と断定すること
