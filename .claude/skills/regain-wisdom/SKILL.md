# REGAIN Wisdom 作成ガイド

## Wisdomとは

AIが処理中の待ち時間に、受講者（スポーツ指導者）へ**思想・フレームワーク・マインドセット**を伝えるコンテンツ。

| 項目 | 内容 |
|------|------|
| 領域 | Mindset / Theory |
| 問い | WHY（なぜやる） |
| 価値 | 思考のOSをアップデートする |
| 効果 | 明日以降も自走できる |

---

## 作成フロー

### 1. コンテキスト確認

```
□ どのWeekに配置するか確認
□ 前後のWeekで何を学んでいるか確認
□ 既存のWisdom資料のトーンを確認
```

参照先：
- `contents/notion-contents/week{N}/wisdom/` - 既存Wisdom
- `docs/Protocol_Wisdom_設計思想.md` - 設計原則

### 2. 構成設計

推奨構成（80〜100行目安）：

```markdown
## 1. 🎯 導入・問題提起（10行）
## 2. 📚 本題・コア概念（20行）
## 3. 🏠 比喩・事例（25行）
   ### ❌ 失敗例
   ### ✅ 成功例
## 4. 🛠️ 実践・対策（20行）
## 5. 💡 まとめ（10行）
```

### 3. 執筆

---

## 設計原則（必ず守る）

### トーン&マナー

| DO | DON'T |
|----|-------|
| 「共有する」スタンス | 「教える」スタンス |
| 先輩からの助言トーン | 講師が教壇から話すトーン |
| カジュアルで親友感覚 | 堅苦しい敬語 |
| 現場の言葉 | 技術用語 |

### 形式ルール

- 絵文字を見出しに使用（🧠 📚 🏠 🛠️ 💡 など）
- ❌ ✅ ⚠️ で対比・注意喚起
- セクション区切りに `---` を使用
- 技術的な数値（トークン数など）は避け、感覚的に説明

### 事例の選び方

```
✅ 日常生活比喩（机、書類、メモ帳、引越しなど）
✅ スポーツ指導の現場感覚（練習メニュー、選手への声かけ）
✅ 受講者が「自分ごと化」できる事例
❌ エンジニア向けの技術例
❌ 抽象的な理論のみ
```

---

## チェックリスト（提出前に確認）

```
□ 「なぜ」が明確に伝わるか？
□ 事例は受講者が自分ごと化できるか？
□ フレームワークは覚えやすい形か？
□ 待ち時間内に伝えられる分量か？（80〜100行）
□ 現場の言葉で書かれているか？
□ 押しつけがましくないか？
□ 既存Wisdomとトーンが統一されているか？
```

## 参考：既存Wisdomの例

### Week1「要求定義の重要性」の構成

```
1. AIは「空気が読めない天才」と思え
2. 事例：「不自由」こそが現場を救う
   - ⭕️ 成功事例：サイゼリヤ
   - ❌ 失敗事例：チャットボット
3. フレームワーク：魔法のCPP
4. 先輩エンジニアからの「転ばぬ先の杖」
```

### Week2「コンテキストの重要性」の構成

```
1. 復習：コンテキスト不足の問題
2. 本題：コンテキストが多すぎても精度が落ちる
3. 日常比喩：机の上がゴチャゴチャ問題
4. 対策：定期的に「お引越し」
5. まとめ：ちょうどいいが最強
```
