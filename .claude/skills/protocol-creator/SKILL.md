# REGAIN Protocol Creator

REGAINプログラムのProtocol（手順書）を新規作成するスキル。

---

## 1. Protocolとは

| 項目 | 内容 |
|------|------|
| **領域** | Action / Execution |
| **役割** | 具体的にどのボタンを押し、どう指示し、どう修正するか |
| **価値** | 迷いを消し、最短距離で成果物を生み出す |

受講者が実際に手を動かしながら体験的な学習をするための**具体的なステップバイステップガイド**。

- 「これをやれば必ずできる」を保証する
- 判断を求めず、実行に集中させる
- プロンプトはコピペ可能な形式で提供

---

## 2. Protocol設計の4原則

### 原則1: 「逃げ道を塞ぐ」設計

REGAINの本質は「成果物が完成すること」。Protocolはそのための強制力。

- **具体的なプロンプトまで書く** → 「何を入力すればいいかわからない」を排除
- **ステップを細分化する** → 「どこから始めればいいかわからない」を排除
- **選択肢を限定する** → 「選べない」を排除

### 原則2: 「考えさせない」設計

- 受講者に判断を求めない
- 判断が必要な箇所は事前にWisdomで教える
- 「やれば動く」を保証する

### 原則3: 「成功体験ファースト」

- まず動くものを見せる → その後に理屈を説明
- エラーが起きにくい設計にする
- 「自分でも作れた！」という感動を最優先

### 原則4: 「再現性」の担保

- 誰がやっても同じ結果になるように書く
- 環境依存の部分は明記する
- ツールのバージョン・設定も必要に応じて記載

---

```markdown

```

[次のProtocolへの誘導、鼓舞する言葉]
```

---

## 4. スタイルガイド

### 絵文字の使用

適切に絵文字を使用して、親しみやすさと視認性を高める:

| 用途 | 推奨絵文字 |
|------|------------|
| 完了・チェック | ✅ |
| 保護・安全 | 🛡️ |
| 開始・実行 | 🚀 |
| 確認・調査 | 🔍 |
| ゴール・目標 | 🎯 |
| 力強さ・頑張り | 💪 |
| 成功・輝き | ✨ |
| ヒント・補足 | 💡 |
| 注意・警告 | ⚠️ |
| Protocol識別 | 🎨 |

### トーン & マナー

- **親しみやすく鼓舞的なトーン**で書く
- 「教える」ではなく「一緒にやる」スタンス
- 「できます！」「大丈夫！」など、安心感を与える言葉を使う

### ターゲットを意識

- スポーツ指導者向けの分かりやすい言葉を使う
- 技術用語は使わない、または必ず補足説明を添える
- 専門用語を使う場合は「〇〇（＝〇〇のこと）」の形式で説明

### プロンプトの書式

- プロンプトはコードブロックで囲み、**コピペ可能**にする
- 言語指定は `markdown` を使用
- 穴埋め部分は `（ここに〇〇を貼り付け）` の形式で明示

## 5. 作成時チェックリスト

Protocol作成完了時に以下を確認:

- [ ] ステップは十分に細分化されているか？
- [ ] プロンプトはコピペ可能な形式か？
- [ ] 受講者に不要な判断を求めていないか？
- [ ] 誰がやっても同じ結果になるか？
- [ ] 成功体験につながる設計か？
- [ ] 技術用語には補足説明が添えられているか？
- [ ] 前後のProtocol/Wisdomとの繋がりが明確か？

## 7. 出力先

作成したProtocolは以下のディレクトリに保存:

```
contents/notion-contents/week{番号}/protocol/[Protocol名].md
```

**ファイル命名規則:**
- 日本語でProtocolの内容を端的に表す名前をつける
- 例: `デモアプリ作成.md`、`要件定義書の作成.md`、`ヒアリングシート.md`
